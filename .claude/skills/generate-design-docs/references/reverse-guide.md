# リバース生成ガイド（reverse-guide）

既存コードから設計書 3 点を復元する際の具体的な手順と書式。

---

## 1. 共通書式

### 1.1 AS-IS ヘッダ（全生成物の冒頭、タイトル直後に必ず挿入）

```markdown
> **本書は既存コードベースからリバース生成された AS-IS 設計書です（generate-design-docs）**
>
> | 項目 | 内容 |
> | -- | -- |
> | 生成日 | YYYY-MM-DD |
> | 対象コミット | `git rev-parse --short HEAD` の結果 |
> | 入力 | docs/00_As_Is/（00_analyze YYYY-MM-DD 実施分）＋ コードベース |
> | 性格 | 「あるべき姿」ではなく「現にそうなっている姿」の記録。変更提案を含まない |
> | 信頼度凡例 | [確認済] = コードで裏取り済 / [推定] = 状況証拠から推測 / [不明] = 読み取り不能 |
```

### 1.2 信頼度の付け方

- **事実（そうなっている）**と**意図（なぜそうした）**を分けて考える。
  事実はコードで裏取りできるので大半が `[確認済]` になる。意図は多くの場合 `[推定]` か `[不明]` になる — それで正しい。
- テンプレートに「選定理由」「目的/KPI」など意図を書く欄がある場合：
  - コミットログ・README・コメントに根拠があれば引用して `[確認済]`
  - 状況から推測できれば `（推定: チーム習熟度と思われる）[推定]`
  - 手掛かりが無ければ `[不明]`（空欄にせず明示する）

### 1.3 適用外セクションの扱い

テンプレートには新規設計向けの章（コスト見積り、将来展望、SLO 目標値など）がある。
リバースで埋められない章は**削除せず**、次のいずれかを記載する：

- 現状の実測・設定値が読み取れる場合 → それを AS-IS として記載
- 読み取れない場合 → `（AS-IS 生成のため対象外。今後の設計活動で定義する）`

章立てを保つことで、後続の設計活動（to-be 化）で同じファイルを育てられる。

## 2. フェーズ 1：機能一覧の復元（最重要）

### 2.1 EP → 機能（FNC）への束ね方

エントリポイント（EP）は実装の口であり、機能（FNC）は**業務目的のまとまり**。1 機能 = 複数 EP が普通。

束ねる手順：

1. `04_Entry_Points.md` の EP を、URL プレフィックス・コントローラ・画面遷移のまとまりでグルーピングする
   （例：`/orders` 配下の画面 3 つ + API 5 つ + 締めバッチ 1 つ → 「受注管理」）
2. グループに**業務目的の名前**を付け、`FNC-001` から採番する
3. 粒度の目安：**`feature-specify` が 1 ユニットとして仕様化できるサイズ**（= 1 回の spec→plan→tasks→implement で消化できる範囲）。迷ったら細かめに割る
4. CRUD 一式は 1 機能にまとめる。承認フロー・帳票出力など業務的に独立したものは分ける

### 2.2 SCR / RPT / IF / BAT の採番

| 05_Feature_List の台帳 | 由来する EP 種別 / 証拠 |
| -- | -- |
| 画面一覧（SCR） | EP 種別 = 画面。SPA の場合はルーター定義の 1 ルート = 1 画面を基本とする |
| 帳票一覧（RPT） | PDF/CSV/Excel を生成・出力するコード（帳票ライブラリ呼び出し、CSV エクスポート処理） |
| 外部 IF 一覧（IF） | ① EP 種別 = API のうち**外部システム向け**のもの ② `06_External_Interfaces.md` の EXT（送信側） |
| バッチ（BAT） | EP 種別 = バッチ / スケジュール / キュー |

- 自システムの画面専用の内部 API は IF に立てず、対応する SCR / FNC の備考に含める（台帳の肥大化を防ぐ）。
- 各行の備考または専用列に由来 ID（`EP-012` / `EXT-003`）と実装位置を必ず書く。

### 2.3 埋め方の注意

- 「目的/KPI」列：業務目的が読み取れなければ `[不明]`。KPI はほぼ確実に `[不明]`（捏造しない）
- 「優先度（MoSCoW）」列：既存機能はすべて稼働中なので `既存（稼働中）` と記載
- 「ステータス」列：`Implemented` を使う（テンプレートのステータス定義に含まれる）
- 「受入条件/要件ID」列：`AS-IS`（現行踏襲）と記載
- 権限列：ミドルウェア・アノテーション・ガード定義から復元し、根拠を付ける
- トレーサビリティ章（7 章）：「機能 → 画面/帳票/IF/バッチ 対応表」と「ロール → 機能/画面 対応表」は必ず埋める。
  「要件 → 機能 対応表」は要件文書が無ければ `AS-IS のため対象外` とする
- CRUD マトリクス（7.4）：`05_Data_Overview.md` の主要エンティティ × FNC で作成する。影響調査の基盤として価値が高いので省略しない

## 3. フェーズ 2：基本設計の復元（04_Basic_Design）

| テンプレート章 | 復元方法 |
| -- | -- |
| システムコンテキスト図 | `06_External_Interfaces.md` の EXT ＋ 利用者ロールから C4 Context を作図 |
| スコープ | 「本システムが現に担っている範囲」を記載（In）。連携先が担う範囲を Out として記載 |
| ADR 表 | フェーズ 3 で作る de facto ADR の要約を転記（先にプレースホルダで良い） |
| 機能構成図 | フェーズ 1 の FNC をサブシステム単位に束ねて Mermaid 化 |
| 主要機能一覧 | 05_Feature_List の FNC サマリ（ID は FNC-xxx をそのまま使う） |
| 非機能要件概要 | **目標値ではなく現状の実装**を書く（例：「冗長化: なし [確認済]」「TLS: あり」）。設計されていない項目は「現状、明示的な設計なし」と記載 |
| 外部インターフェース概要 | EXT 一覧の要約 |
| 運用・保守の基本方針 | CI/CD 定義・監視設定・ログ実装から読み取れる現状のみ |

## 4. フェーズ 3：アーキテクチャ設計の復元（03_Architecture）

### 4.1 de facto ADR の書き方

「決定の記録」ではなく「**発掘された決定**」。テンプレートの ADR 形式に合わせ、次の流儀で書く：

```markdown
# ADR-001: PostgreSQL の採用（de facto）
* Status: Accepted（既成事実）
* Date: 不明（初回コミット 2019-04 時点で既に採用 [確認済: git log]）
* Decision Makers: 不明

## Context
採用理由の記録は見つからなかった。（推定: 当時の標準構成 [推定]）

## Decision
主データストアとして PostgreSQL を使用している。バージョン 15.x [確認済: docker-compose.yml]

## Consequences
- 現状: マイグレーション 120 本、PostgreSQL 固有機能（JSONB）を 8 箇所で使用 [確認済]
- 変更コスト: 高（固有機能の使用箇所と全マイグレーションの書き換えが必要）
```

ポイント：**Consequences に「変更コスト」を必ず書く**。追加開発の意思決定で最も使われる情報になる。

### 4.2 ADR に起こす対象の目安

- 言語・FW・DB などの主要スタック（各 1 本）
- アーキテクチャスタイル（モノリス/マイクロサービス、レイヤ構成）
- 認証・認可方式
- 非自明な選定（自作 FW、古い技術の継続使用、二重実装など）— これらは特に変更コストの記録価値が高い

### 4.3 その他の章

| テンプレート章 | 復元方法 |
| -- | -- |
| アーキテクチャスタイル | `03_Code_Structure.md` の判定結果。「採用理由」は 1.2 の信頼度ルールで |
| 技術スタック表 | `02_Tech_Stack.md` から転記。「選定理由」列は `[不明]`/`[推定]` を許容、「更新方針」列は現状観察（例：3 年更新なし [確認済]） |
| C4 図・構成図 | Context は 04 と共通。Container は実行単位（Web/ワーカー/DB 等）を docker-compose・インフラ定義から復元 |
| 非機能要件の実装方式 | 現状実装のみ（可用性: 単一インスタンス、監視: なし 等）。**改善提案を混ぜない** |
| エラーハンドリング方針 | 実装済みの共通ハンドラ・レスポンス形式を実コードから引用 |
| CI/CD | CI 定義ファイルから実フローを Mermaid 化 |
| コスト見積り | `（AS-IS 生成のため対象外）` |

## 5. 既存 docs/ との衝突時の選択肢（フェーズ 0 ゲートで提示）

出力先に既存ファイルがある場合、AskUserQuestion で以下を提示する：

1. **上書き**（既存が古い調査メモ等で、破棄してよい場合）
2. **別名保存**：`docs/01_Project_Design/as_is/` 配下に出力（既存の to-be 設計と併存させる場合）
   - この場合、`feature-specify` が両方を読む可能性があることをユーザに伝える
3. **中断**（既存文書の扱いを先に整理する）

Git 管理下であっても、無確認の上書きはしない。
