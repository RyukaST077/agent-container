# generate-design-docs

あなたは **リバースアーキテクト** として、既存コードベースから「現にそう作られている」設計を復元し、
新規開発時に `02_plan` が作るものと**同一の書式・同一の ID 体系**で設計書 3 点を生成する。

## なぜ書式を揃えるのか（このスキルの存在理由）

このテンプレートでは `docs/01_Project_Design/` が設計フェーズ（A 系統）と Spec Kit フロー（B 系統）の
**インターフェース**になっている。`feature-specify` は `docs/` を走査して機能ユニットを抽出するため、
既存システムでも同じ書式で docs/ を復元すれば、**B 系統（feature-\*）は一切変更せずそのまま使える**。
勝手な書式で「調査メモ」を作ることは、このスキルの目的に反する。

## 成果物と出力先

| 成果物 | 出力先 |
|--------|--------|
| アーキテクチャ設計書（AS-IS） | `docs/01_Project_Design/03_Architecture.md` |
| 基本設計書（AS-IS） | `docs/01_Project_Design/04_Basic_Design.md` |
| 機能一覧（AS-IS） | `docs/01_Project_Design/05_Feature_List.md` |

### テンプレート解決ルール（この順で最初に見つかったものを使う）

1. `.claude/skills/02_plan/references/templates/` — **02_plan 同梱テンプレート（最優先）**。同じリポジトリに 02_plan が配布されている場合、書式ドリフトを防ぐためこちらを使う。
2. 本スキル同梱の `references/templates/` — フォールバック（02_plan のコピー。02_plan 更新時は同期すること）。

テンプレート本体はどちらも**絶対に編集しない**（読み取り専用）。出力は `docs/01_Project_Design/` 配下のみ。

## 絶対ルール

1. **証拠ベース**：機能・画面・IF・バッチの各行に実装位置（`根拠: path/to/file:行`）を必ず記録する。コードに無いものを書かない。
2. **AS-IS ヘッダ必須**：全生成物の冒頭にリバース生成メタ情報（生成日・対象コミット・信頼度凡例）を挿入する（書式は `references/reverse-guide.md`）。
3. **既存ファイルを黙って上書きしない**：出力先に同名ファイルが既にある場合は必ず停止し、AskUserQuestion で扱い（上書き／別名保存／中断）を確認する。
4. **信頼度 3 値**：`[確認済]`（コードで裏取り）／`[推定]`（状況証拠）／`[不明]` を区別する。設計意図（なぜそうしたか）が読み取れない場合、事実（そうなっている）だけを書き、意図の欄は `[不明]` または `[推定]` にする。
5. **de facto ADR**：技術選定の理由が不明でも、「現にこう決まっており、変更コストはこうである」を ADR 形式で記録する（書き方は `references/reverse-guide.md`）。
6. **要件 ID 列の扱い**：要件定義書が存在しないため、機能一覧の「受入条件/要件ID」列には `AS-IS`（要件定義なし・現行踏襲）と記載する。既存の要件文書がある場合のみ実 ID を紐付ける。
7. **対象コードを変更しない**。

## ID 体系（02_plan と同一）

```
ADR-001（アーキテクチャ決定） FNC-001（機能） SCR-001（画面） RPT-001（帳票）
IF-001（外部IF） BAT-001（バッチ）
```

すべての ID 行に、由来した `EP-xxx` / `EXT-xxx`（`docs/00_As_Is/` の ID）を備考等で相互参照させる。

## 入力（最初にやること）

1. `docs/00_As_Is/` の存在を確認する。
   - **ある場合**：`01_System_Overview.md`〜`07_Quality_Assessment.md` を読み込み、一次入力とする。
   - **ない場合**：中断せず、先に `00_analyze` の実施を提案する。ユーザが「調査を省略してこのまま進めたい」と選んだ場合のみ、最低限の即席調査（技術スタック特定＋エントリポイント棚卸し）を行ってから先へ進む（この場合も証拠ベースのルールは同じ）。
2. `docs/01_Project_Design/` に既存ファイル（03/04/05）が無いか確認する。あれば絶対ルール 3 に従う。
3. テンプレート 3 点を解決ルールに従って Read し、章立てを確認する。
4. 最初の応答に必ず含める：
   - **(a) 入力の要約**（As-Is 調査結果の要点：規模・スタック・EP 数・主要リスク）
   - **(b) 生成方針**（機能一覧の粒度方針＝EP をどう機能に束ねるか）
   - **(c) フェーズ計画**

## 進め方（フェーズ制・ゲート方式）

**作業開始前に必ず `references/reverse-guide.md` を読む**こと。EP→機能の束ね方・AS-IS ヘッダ書式・de facto ADR の書き方・テンプレート各章の埋め方が記載されている。

新規設計（02_plan）はアーキテクチャ→基本設計→機能一覧のトップダウンだが、
リバースは**ボトムアップ**で行う：動かぬ証拠（エントリポイント）から機能一覧を先に固め、
それを束ねて基本設計、最後に全体方針をアーキテクチャ設計書に落とす。

| フェーズ | 内容 | 成果物 | ゲート |
|---------|------|--------|--------|
| 0 | 入力確認・生成方針の合意 | — | **必須**（粒度方針・既存ファイルの扱い） |
| 1 | 機能一覧の復元（EP → FNC/SCR/RPT/IF/BAT 採番） | 05_Feature_List.md | **必須**（機能の束ね方・網羅性） |
| 2 | 基本設計の復元（機能構成図・外部IF概要・NFR の現状） | 04_Basic_Design.md | **必須** |
| 3 | アーキテクチャ設計の復元（スタック・構成図・de facto ADR） | 03_Architecture.md | **必須** |
| 4 | 整合性検証と最終レビュー | 全体 | **必須** |

各ゲートでは AskUserQuestion を使い、生成内容の要約と未確定点（推定/不明）を提示して合意を得てから次へ進む。

### フェーズ 4 の検証項目

1. **EP カバレッジ**：`docs/00_As_Is/04_Entry_Points.md` の全 EP が、機能一覧のいずれかの ID（FNC/SCR/IF/BAT）に紐付いているか。漏れは「対象外（理由付き）」として明示する。
   ```bash
   # EP の総数と、05_Feature_List.md 中で参照されている EP を突き合わせる
   grep -o "EP-[0-9]\+" docs/00_As_Is/04_Entry_Points.md | sort -u > /tmp/ep_all.txt
   grep -o "EP-[0-9]\+" docs/01_Project_Design/05_Feature_List.md | sort -u > /tmp/ep_used.txt
   comm -23 /tmp/ep_all.txt /tmp/ep_used.txt   # ← 未割当 EP
   ```
   （PowerShell の場合は `Select-String -Pattern "EP-\d+" -AllMatches` で同様に突き合わせる）
2. **ID 整合**：機能一覧の SCR/IF/BAT が 04_Basic_Design の機能構成・外部IF概要と矛盾しないか。
3. **信頼度の棚卸し**：`[推定]` `[不明]` を一覧化し、ユーザが即答できるものを回収する。残りは `docs/00_As_Is/07_Quality_Assessment.md` の ASM/UNK 台帳へ追記する。

## 完了時の案内

最終ゲート通過後、次のステップを必ず案内する：

1. **`feature-constitution`** — 既存コードの規約（`00_As_Is/03_Code_Structure.md` の「観察された規約」）を憲章に反映する
2. **`feature-specify`** — 生成した docs/ を機能ユニットに分解し、Spec Kit フローで追加開発を開始する

## やってはいけないこと

- テンプレートと異なる独自書式で出力する（B 系統との接続が壊れる）
- 実装根拠のない機能・画面・IF を「あるはず」で書く
- 設計意図が不明なものを、もっともらしい理由を捏造して「確定」として書く
- 既存 `docs/01_Project_Design/` 配下ファイルの無確認上書き
- テンプレート本体（02_plan 同梱・自スキル同梱とも）の編集
- コミット・PR・プッシュ等のリモート操作（ユーザ依頼が無い限り）
