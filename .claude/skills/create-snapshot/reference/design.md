# v1 API スナップショット回帰テスト 設計解説

運用手順は [snapshot-test.md](./snapshot-test.md) にまとめている。
本書は「**スナップショットに何を記録しているか**」「**なぜこれで機能の保証になるのか**」「**どこまで保証しないか**」を整理する、設計の説明書である。

関連ソース:
- `tools/v1-test/v1_test_runner.sh` … 実 API を curl で叩き、JSONL でレスポンスをキャプチャする
- `tools/v1-test/v1_test_with_snapshot.sh` … キャプチャを `snapshots/responses-<RUN_TAG>.json` に変換し、マスターと照合する
- `tools/v1-test/v1_snapshot_comparator.py` … 正規化と差分検出、Markdown レポート生成
- `tools/v1-test/lib/assert.sh` … レスポンス値のアサーション（Layer 1）
- `.claude/hooks/snapshot-check.sh` … Claude Code Stop hook で自動検証
- `snapshots/responses-master.json` … ゴールデンマスター（Git 管理、承認ベース）

---

## 1. 全体像：3 層で機能を押さえる

スナップショット比較は単独のテストではなく、以下 3 層の一部である。Layer 1/3 は curl ベースでフレームワーク非依存のため、Laravel のバージョンアップ等で壊れにくい。

| Layer | 目的 | 実装 | 壊れやすさ |
|---|---|---|---|
| 1 | レスポンス値・型アサーション（HTTP コード、必須フィールド、型） | `v1_test_runner.sh` + `lib/assert.sh` | 低（curl + python3） |
| 2 | アプリ内部のユニット/結合テスト | PHPUnit / Laravel Test | 高（FW 依存） |
| 3 | レスポンス全体のスナップショット比較（回帰検知） | `v1_test_with_snapshot.sh` + `v1_snapshot_comparator.py` | 低 |

Layer 3 の役割は「**外部から見える API の振る舞いが、承認された時点から意図せず変わっていないこと**」を面で保証することにある。Layer 2 が「書いたコードが書いた通りに動くか」を見るのに対し、Layer 3 は「既存利用者にとって壊れていないか」を見る。

## 2. スナップショットに記録している内容

`snapshots/responses-*.json` は、1 回のテスト実行で叩いた各 API コールを配列にしたもの。1 エントリは下記のキーを持つ。

| キー | 内容 | 照合で使うか |
|---|---|---|
| `timestamp` | 実行時刻（ISO 8601） | × 比較対象外 |
| `suite` | テストスイート名（`normal` など） | △ 参考表示 |
| `endpoint` | エンドポイント識別子（例: `languages`, `terms/version`） | ○ マッチングキー |
| `method` | HTTP メソッド | ○ マッチングキー |
| `url` | 実リクエスト URL（クエリ込み） | △ 参考表示 |
| `http_code` | レスポンスの HTTP ステータスコード | ○ 差分検出対象 |
| `expected` | 期待 HTTP コード（runner が想定したもの） | × 比較対象外 |
| `result` | `OK` / `NG` の判定 | × 比較対象外 |
| `response_body` | レスポンスボディ（JSON を再帰的に保持） | ○ 差分検出対象（正規化後） |

### マッチングキー

`v1_snapshot_comparator.py` は `(endpoint, method)` の組でマスターと現行を突き合わせる (`_get_key`)。`url` や `suite` はキーに含めない。これは、クエリパラメータのゆらぎ（後述の正規化でも吸収しきれないもの）で別エンドポイント扱いになるのを避けるため。

### キャプチャ方法

`v1_test_runner.sh` 側が環境変数 `SNAPSHOT_CAPTURE_FILE` を受け取り、各 curl 実行の結果を JSONL で 1 行ずつ書き出す。`v1_test_with_snapshot.sh` がそれを読み直して JSON 配列に整形する（ログ再パースはフォールバック）。テストは実 API を実 DB 付きで叩く＝**モックを介さない**ので、DI 層の食い違いや ORM の生成 SQL 変化も最終的なレスポンスに出てくれば検知できる。

## 3. 正規化：「壊れてないのに毎回差分」を起こさない工夫

生のレスポンスをそのまま比較すると、タイムスタンプ・ID・URL・ランダムフィクスチャ値などで毎回差分が出てしまい、実質的に無意味になる。`v1_snapshot_comparator.py` は比較前にレスポンスボディを再帰的に正規化する (`_normalize_dict` → `_normalize_value`)。

### キー名ベースの正規化

| キー名パターン | 置換後トークン |
|---|---|
| `id` / `*_id` / `*_id_*` | `{{ID}}` |
| `created_at` / `updated_at` / その他 `*_at` | `{{TIMESTAMP}}` |
| `token` / `secret` / `password` / `signature` / `api_key` / `receipt` を含む | `{{SECRET}}` |
| `code` / `qrcode` / `*_code` | `{{CODE}}` |
| `url` / `uri` / `link` / `image` / `thumbnail` / `avatar` で `http(s)://` | `{{URL}}` |
| `nick_name` で `v1-runner-user-*` | `{{TEST_NICKNAME}}` |
| `name` / `description` でテストフィクスチャ書式 | `{{TEST_TEXT}}` |

### 値パターンベースの正規化

| 値の形 | 置換後トークン |
|---|---|
| `YYYY-MM-DDTHH:MM:SS` | `{{TIMESTAMP}}` |
| `YYYY-MM-DD` / `YYYY/MM/DD` | `{{DATE}}` |
| UUID | `{{UUID}}` |
| 32〜64 文字の HEX | `{{HEX_DIGEST}}` |
| 8 桁以上の数字 | `{{NUMERIC_ID}}` |
| `v1-runner-...-YYYYMMDD-HHMMSS-N[-updated]` | `{{TEST_FIXTURE}}` |
| `no-such-product-*` | `{{PRODUCT_ID_INVALID}}` |
| `spot_index_*_*_xx` / `area_index_*_*_xx` | `{{CACHE_KEY}}` |
| `http(s)://...` | `{{URL}}` |

### エンドポイント単位のボディ比較スキップ

`info` エンドポイントはビルド時刻等の揺らぎが大きすぎるため、`_should_ignore_body_compare()` で本文比較を省略し HTTP コードのみ見る。

### 正規化の立ち位置

正規化は「**意味のある差分だけを残すフィルタ**」である。裏返すと、**正規化で潰された値の変化は検知できない**。タイムスタンプや ID の順序・有無そのものが仕様である場合（例: ソート順が ID 昇順であること）は、本レイヤーでは担保されず Layer 1 のアサーション側で書く必要がある。

## 4. 差分判定とレポート

比較は 4 種類を出す。

| 種別 | 意味 | 終了コード |
|---|---|---|
| `NEW` | 現行にだけ存在するエンドポイント | 失敗 (1) |
| `REMOVED` | マスターにだけ存在するエンドポイント | 失敗 (1) |
| `CHANGED` | 同キーで `http_code` か正規化後 `response_body` が変化 | 失敗 (1) |
| `CONTRACT_ERROR` | エントリが最低契約（`endpoint`/`http_code` + `result` か `method`）を満たさない | 失敗 (1) |

`CONTRACT_ERROR` があるのは、スナップショット JSON 自体のスキーマが将来変わっても「壊れて形式不一致になった状態」を即検知できるようにするため。新旧スキーマの混在（`suite`/`url` が無い旧版）は許容するが、最低限の形は割らないようにしている。

差分は `snapshots/responses-<RUN_TAG>-report.md` に Markdown で書き出される。`CHANGED` のボディ差分は unified diff 形式で先頭 20 行まで、それ以上は省略表示。

## 5. ワークフロー：ゴールデンマスター方式

1. **承認済みマスター** = `snapshots/responses-master.json` を Git で管理する（人がレビューして承認した状態）。
2. 各テスト実行で `responses-<RUN_TAG>.json` を新規生成し、マスターと比較する（`test` アクション）。
3. 差分ゼロなら PASS。差分があれば FAIL し、レポートを人がレビューする。
4. レビュー結果が「仕様変更で正しい」ならば `approve` で新マスターに昇格させ、Git コミット＝**変更が明示承認される**。
5. 「バグ」ならばコードを直して再実行。

ここで重要なのは、**マスター更新が必ず人間のレビューを経由するコミットであること**。回帰テストの強度は「マスターの正しさ」に等しく、これをコードレビューと同じゲートに載せることで、「何となく差分を全部 approve する」を抑止している。

### Stop hook による自動実行

`.claude/hooks/snapshot-check.sh` が Claude Code の Stop フックに仕込まれており、会話終了時に自動で `v1_test_with_snapshot.sh test` が走る。失敗すれば `decision: block` を返し、Claude に修正を継続させる。つまり**編集 → 自動回帰検証 → 差分があれば即ブロック**というループが常時回っている。

## §5.5 DB リセットとフィクスチャ管理

### なぜ DB リセットが必要か

スナップショットテストは「同じ入力に対して同じ出力」を保証する。しかし API レスポンスは DB の状態に依存するため、DB が異なれば同じリクエストでも異なるレスポンスが返る。

record のたびに DB を既知状態に戻さないと:
- 前回の record で作成したテストユーザーが SoftDeletes で見えなくなる
- フィクスチャ ID のレコードが削除済みで 404 になる
- シリアルコード等の「一度だけ使える」リソースが使用済みになる

### reset-db.sh の設計

`reset-db.sh` は 6 ステップで構成される:
1. DB リセット（migrate:fresh 等）
2. シーダー実行
3. 共通フィクスチャ投入（API キー等）
4. プロジェクト固有フィクスチャ投入
5. テストユーザー登録（API 経由）
6. fixture.env 書き出し

冪等性が重要: 何度実行しても同じ初期状態になること。

### fixture.env パターン

seed 結果の動的 ID（テストユーザー ID 等）を `fixture.env` に書き出し、`capture.sh` が起動時に source する。これにより:
- シーダーの変更で ID が変わっても capture.sh の修正が不要
- ランダム生成される user_id 等を自動連携できる
- 環境変数として export されるので、capture.sh 内で `${FIXTURE_USER_ID}` のように参照できる

### SEED_CMD による自動化

`snapshot.sh` の `SEED_CMD` 環境変数を設定すると、`record` / `test` アクション実行前に自動で DB リセットが走る:

```bash
export SEED_CMD="bash tools/snapshot/reset-db.sh"
export CAPTURE_CMD="bash tools/snapshot/capture.sh"
bash tools/snapshot/snapshot.sh test   # reset-db.sh → capture.sh → 比較
```

### ビジネスロジック上の罠

シーダーでデータが入っても API から見えないケースが頻出する。代表例:
- **publish_scope / enabled**: シーダーのデフォルトが「非公開」→ API は 404
- **SoftDeletes**: capture.sh 内の DELETE で論理削除 → 後続テストが失敗
- **ID エンコーディング**: 内部 ID と API 上の ID が異なる（暗号化 / UUID）
- **外部サービス連携**: 決済 API 等はローカルで 500 が正常（ベースラインとして固定化）

## 6. なぜこれで機能を保証できるか

この仕組みは「実装の正しさ」ではなく「**観測可能な API 契約の不変性**」を保証する。根拠は以下。

1. **実 API + 実 DB で叩く**: ランタイム・DI・ORM・ミドルウェア・ルーティング・シリアライザ・DB マイグレーションまで含めた完成系の出力を比較する。どこか 1 層の変更でも最終レスポンスに漏れ出れば検知される。
2. **HTTP コードとレスポンスボディを両方固定**: 成功/失敗の切り替わり（200→500 や 200→422）と、ボディ構造・値の変化の双方をカバーする。
3. **正規化で「意味のある差分」だけを残す**: ノイズで黙殺されないので、差分が出た＝仕様変更 or バグの二択になる。人のレビュー判断が効く状態を保てる。
4. **マスター更新が明示承認 + Git コミット**: 「知らないうちに仕様が変わっていた」が成立しない。`snapshots/responses-master.json` の diff がそのまま「外から見た API 変更履歴」になる。
5. **フレームワーク非依存**: curl + python3 のみで走るため、Laravel 9.5→12 のようなメジャーバージョンアップ中でも Layer 3 は壊れない。アップグレードでアプリ内部テスト (Layer 2) が一時的に落ちても、Layer 3 は「ユーザー体験は変わっていない」ことだけを見続けられる。
6. **Stop hook による常時実行**: 手動で忘れる余地が無い。差分があれば Claude 側の会話が止まるため、差分を無視したまま作業が進行しない。

まとめると、スナップショット比較は「API 利用者から見た振る舞いを固定化し、そこからの逸脱をすべて人間のレビューに押し戻す装置」として機能する。

---

## 7. 保証しないこと（限界）

- **内部実装の正しさ**: 同じレスポンスを返せてしまえば内部でどれだけ非効率/不健全でも通る。ユニット/結合テストで別途担保する。
- **正規化で潰された値の妥当性**: `{{TIMESTAMP}}` が本当に「実行直後の時刻か」、`{{ID}}` が「自動採番の連番か」といった意味論は見られない。必要なら Layer 1 のアサーションに落とす。
- **マスター自体の正しさ**: 誤った動作を承認してしまえばそれが基準になる。初期マスター確定と `approve` の判断はコードレビュー品質と同じ重みがある。
- **非決定的な並び順**: JSON 比較は `sort_keys=True` だが、配列の順序はそのまま比較される。順序がマスター固定の API で実装側の順序が揺れれば差分になるので、固定ソートがない API は Layer 1 でソート保証するか、マスター側を安定化する必要がある。
- **未呼び出しのエンドポイント**: `v1_test_runner.sh` が叩かないエンドポイントは一切保護されない。網羅率はランナーの実装範囲で上限が決まる。
- **テスト実行環境固有の差分**: 外部依存（決済・通知・地図 API 等）のレスポンスが本番と差がある場合、スナップショットも差分を含む。モック化するかフィクスチャ化する判断が別途必要。

---

## 8. 「機能の保証」と言ったときの意味

本仕組みが明言できる命題は次のレベル:

> **承認された時点のマスターに対して、現行の実装は同じ入力に対して同じ HTTP コードと（正規化後で同一の）レスポンスボディを返す。**

逆に言えば、この命題が壊れるような変更は人間のレビュー（=マスター更新コミット）を必ず通る。新規機能は新しいエンドポイントとしてマスターに追加され、仕様変更は diff として履歴に残り、回帰（意図しない変化）は CI または Stop hook で失敗する。

この一貫性こそが「機能の保証」であり、**個々のテストケースが網羅的である保証ではなく、変化が必ず見える化される保証**であると理解するとよい。
