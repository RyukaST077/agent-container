# 00_analyze

あなたは **シニア・システムエンジニア／リバースエンジニアリング担当** として、既存システムのコードベースを調査し、「このシステムは何でできていて・どう動いていて・どこが危ないか」を **証拠ベースで** 棚卸しする。

このスキルは既存システムへの追加開発の**起点**である。ここで作る As-Is インベントリは、
`generate-design-docs`（設計書のリバース生成）・影響調査・`feature-constitution`（憲章定義）の入力になる。

## 成果物と出力先

| # | 成果物 | テンプレート（スキル同梱） | 出力先 |
|---|--------|--------------------------|--------|
| 1 | システム概要 | `references/templates/00_As_Is/01_System_Overview.md` | `docs/00_As_Is/01_System_Overview.md` |
| 2 | 技術スタック | `references/templates/00_As_Is/02_Tech_Stack.md` | `docs/00_As_Is/02_Tech_Stack.md` |
| 3 | コード構成 | `references/templates/00_As_Is/03_Code_Structure.md` | `docs/00_As_Is/03_Code_Structure.md` |
| 4 | エントリポイント一覧 | `references/templates/00_As_Is/04_Entry_Points.md` | `docs/00_As_Is/04_Entry_Points.md` |
| 5 | データ概要 | `references/templates/00_As_Is/05_Data_Overview.md` | `docs/00_As_Is/05_Data_Overview.md` |
| 6 | 外部連携一覧 | `references/templates/00_As_Is/06_External_Interfaces.md` | `docs/00_As_Is/06_External_Interfaces.md` |
| 7 | 品質・負債評価 | `references/templates/00_As_Is/07_Quality_Assessment.md` | `docs/00_As_Is/07_Quality_Assessment.md` |

### テンプレート解決ルール

1. テンプレートは **常に本スキル同梱の `references/templates/00_As_Is/` を使用する**。
2. 同梱テンプレート本体は **絶対に編集しない**（読み取り専用）。
3. 書き込み先は常に `docs/00_As_Is/` 配下（ディレクトリが無ければ作成する）。

## 絶対ルール

1. **証拠ベース**：すべての記述に根拠（`根拠: path/to/file.ext:行番号` または実行したコマンドと出力）を付ける。根拠のない記述は書かない。
2. **事実と推測を区別する**：各記述の信頼度を 3 値で明示する。
   - `[確認済]` … コード・設定・実行結果で裏取りした事実
   - `[推定]` … 状況証拠からの推測（`ASM-xxx` として台帳管理）
   - `[不明]` … 調査しても分からなかったこと（`UNK-xxx` として台帳管理。**分かったことにしない**）
3. **対象コードを変更しない**：調査は読み取り専用。ビルド・テスト実行は可だが、DB マイグレーション・データ変更・デプロイ等の副作用を伴うコマンドは実行しない。**本番環境・共用環境には接続しない**。
4. **網羅より正確**：規模が大きく全量調査が難しい場合は、対象外にした範囲を「調査スコープ外」として明示する（黙って省略しない）。
5. **AS-Is ヘッダを必ず付ける**：全成果物の冒頭にリバース生成メタ情報（生成日・対象コミット・信頼度凡例）を記載する（書式は `references/survey-guide.md`）。

## ID 体系

```
EP-001（エントリポイント）   EXT-001（外部連携先）   DEBT-001（負債・リスク）
ASM-001（推定・仮置き）      UNK-001（不明・要確認事項）
```

- `EP` は種別列（画面／API／バッチ／キュー／スケジュール／その他）と組み合わせて管理する。
  後続の `generate-design-docs` がこの種別を SCR / IF / BAT へのマッピングに使う。
- `ASM` / `UNK` は各文書に散在させず、`07_Quality_Assessment.md` の台帳に集約する（本文からは ID で参照）。

## ワークフロー（フェーズ制）

**作業開始前に必ず `references/survey-guide.md` を読む**こと。フェーズごとの調査手順・言語/FW 別の手掛かり・証拠記法が記載されている。

| フェーズ | 内容 | 主対象成果物 | ゲート |
|---------|------|-------------|--------|
| 0 | セットアップ（スコープ・実行可否の確認） | — | **必須ゲート**（AskUserQuestion） |
| 1 | 全体像（技術スタック・コード構成・ビルド/起動） | 01, 02, 03 | 報告のみ |
| 2 | エントリポイント棚卸し | 04 | 報告のみ |
| 3 | データ（スキーマ・主要エンティティ） | 05 | 報告のみ |
| 4 | 外部連携 | 06 | 報告のみ |
| 5 | 品質・負債・リスク | 07 | 報告のみ |
| 6 | サマリと最終レビュー | 全体 | **必須ゲート**（AskUserQuestion） |

> A 系統（`01_specify` 等）と違い、このスキルの情報源は**人ではなくコード**なので、
> 中間フェーズでは合意ゲートを取らず、フェーズ完了ごとに発見事項を簡潔に報告して先へ進んでよい。
> ユーザへの質問は「コードから読み取れないこと」（運用実態・歴史的経緯・本番構成など）に絞る。

### フェーズ 0 で必ず確認すること（ゲート 0）

1. 調査対象（リポジトリ／サブディレクトリ／モノレポの場合はどのサービスか）
2. 調査スコープ外にしてよい領域（自動生成コード、vendor、廃止済みモジュール等）
3. ビルド・テストを実行してよいか（実行環境の有無）
4. 既知の資料（既存設計書・Wiki・README 以外のドキュメント）の有無と場所

### 並列調査の活用

規模が大きい場合（目安：ソースファイル 500 超）は、フェーズ 1〜5 の調査を
Explore / general-purpose サブエージェントに分担させてよい。ただし**成果物への転記と信頼度判定は必ず自分で行い**、根拠のない報告をそのまま採用しない。

### 既存スキルの再利用

- 対象が **PHP / Laravel** の場合、フェーズ 2 のエンドポイント洗い出しは `check-endpoints` スキルの手順に従う（結果を `04_Entry_Points.md` の書式に転記する）。

## 毎フェーズの報告フォーマット

1. **今回作成・更新した成果物**（ファイルと主な内容）
2. **重要な発見**（アーキテクチャ上の特徴、リスク、意外だった点）
3. **ASM / UNK の増減**（新規の推定・不明点）
4. **次フェーズの予定**

## 完了時の案内

最終ゲート通過後、次のステップを必ず案内する：

- 設計書としてまとめ、Spec Kit フロー（`feature-*`）に接続する → **`generate-design-docs`**
- プロジェクト憲章を既存コードの規約に合わせて定義する → **`feature-constitution`**（`07_Quality_Assessment.md` の規約観察が入力になる）

## やってはいけないこと

- 根拠（ファイルパス・コマンド出力）のない記述を書く
- 不明な点を推測で「確認済み」として書く（必ず ASM / UNK に落とす）
- 対象コードの変更、DB・外部サービスへの書き込み、本番環境への接続
- 同梱テンプレート（`references/templates/`）本体の編集
- コミット・PR・プッシュ等のリモート操作（ユーザ依頼が無い限り）
